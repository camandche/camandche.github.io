Command Line Primer

In a couple of sentences, how would you describe the command line in plain English? Can you think of an analogy for it?

The Command Line Interface (CLI) is a text program terminal that interprets lines of code & follows the instructions contained within to create, manipulate, store & delete content within a computing system. It is also faster than using the Graphic User Interface (GUI). It's like having to instruct a robot to walk around. You have to think about writing the code for it in a specific order, like to open it's eyes, stand up from it's resting spot, then move it's left arm forward at the same time that the right leg moves forward & then repeat that process but swap it around where the right arm moves forward when the left leg moves forward & have that repeating on a loop until you want to give it another command. If you enter in the wrong set of instructions at the wrong time, your output will not be what you desired, but, it does interpret things exactly as you programmed it!

Name 5 commands you used, and what they do

  1. cd = change directory
  2. pwd = print working directory
  3. git add command as in 'git add -A' = (-A flag = add all content). 3 part process, 1 = STAGING - gets content ready; takes files you made changes to locally in the Working Directory & stores them in the Staging Index, like a waiting room

Install and Explore Git

Whats the difference between git and GitHub?

Git is a version control tool that's stored locally on your machine. It allows the user to clone or fork a repository from GitHub (a Distributed Version Control System (DVCS)) - which is the remote repository stored up in the cloud). Git allows a user to manipulate the content within their own repository without making changes to the remote repository until the push/pull commands are used. GitHub allows multiple users to access & clone their own branches or forks of a repository & work on the same code at the same time as others while GitHub will use version control to track different changes & document all amended versions. It removes the need to store multiple versions of the same code in various places which could lead to lost data & time.

Can you think of an analogy to describe them?

Having an alternate timeline different to the original path is like two best mates that eventually go their own separate ways in life. They may cross paths again one day (clone), or maybe never againa (fork).

Branch, Pull, Merge

What is Master?

This is the default branch where the best-tested code is kept & (in this instance), is stored upstream online in GitHub. This repository can be cloned or forked to Git for the local user to manipulate the content & will need to go through a 3-step process using the push/pull commands to send it back upstream

Why create a Branch?

A branch repository is a copy taken from the master branch to be used to manipulate content within the repository without overriding the content held in the Master branch

How would you describe stage and commit to your non-tech-savvy friend?

The Staging Index is like a waiting room (just like being engaged). You're putting your content (files/directories etc) in this room waiting for you to send the package to the repository. Once you are ready to send the files here, this is where we commit to the process (or in this instance, marry your loved one)

GitHub Fork & Clone

What are some examples of when you would fork?

Fork is where you're taking a copy of a project from the Master repository & create a new upstream repository (which remains in the cloud in GitHub). Examples of this may be to start a new project that may not be related to the original Master repository or that you do not have write permission to work on the main repository directly.
